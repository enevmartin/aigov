# CLAUDE.md — Конституция на aigov.bg

Ти работиш по aigov.bg — AI електронно правителство на България. Министерствата са AI агенти,
които събират публични данни, анализират ги и публикуват отчети и дашборди на български.
Този файл е върховният закон на repo-то. При конфликт между инструкция в чата и този файл —
този файл печели, освен ако човекът изрично не каже, че променя конституцията.

## Чл. 1 — Архитектурни инварианти (ненарушими)

1. `core/` НИКОГА не импортира от `brains/` и не съдържа код, специфичен за LLM доставчик.
   Единствената връзка ядро↔мозък е файловият договор за задачи (`core/contracts`).
2. `site/` чете САМО `published/`. Без API, без бази, без runtime зависимости от ядрото.
3. `ministries/` съдържа САМО декларации (YAML + Markdown промпти). Нула изпълним код.
4. Мозъкът се сменя от `config.yaml` → `brain:`. Всеки нов runtime е нов адаптер в `brains/`,
   имплементиращ `brains/base.py::BrainAdapter`. Нищо друго не се пипа при смяна.
5. Мозъците НИКОГА не пишат в `published/` — само `core/publish` след валидация на схемите.
6. Сурови данни (`data/raw/`) са ефимерни: обработват се и се трият. Съхраняват се само
   агрегати и отчети. `data/` е git-ignored.

Ако задача те изкушава да нарушиш член 1 — спри и попитай човека, преди да пишеш код.

## Чл. 2 — Git дисциплина

- След ВСЯКА завършена задача: `git add -A && git commit -m "<описателно, английски>" && git push`.
- Един комит = една логическа задача. Не трупай.
- Никога не комитвай: секрети, `.env`, `data/`, `node_modules`, генерирани артефакти извън `published/`.
- Ако push се провали — оправи го веднага, не продължавай към следваща задача.

## Чл. 3 — Договорът за задачи

- Задача: `tasks/pending/{id}/` с `task.yaml`, `input/`, `expected.schema.json`.
- Изход: `output/report.md` (+ front-matter), `output/aggregates.json`, при новини `output/news.json`.
- Преместванията между `pending/ → running/ → done|failed/` са atomic rename.
- Всяка промяна по схемите в `core/contracts` изисква: миграционна бележка в комита
  и обновяване на `tests/` + `_template` министерството в същия комит.

## Чл. 4 — Министрите

- Всеки министър е `ministries/{slug}/ministry.yaml` + `prompts/`. Ново министерство се
  създава чрез копиране на `_template` — никога на ръка с отклонения от схемата.
- Отчетите са на книжовен български, с цитирани източници (URL + дата на извличане).
- Тон: компетентен, спокоен, аналитичен. Без политически пристрастия, без сензации.

## Чл. 5 — Правни и етични предпазители

- НИКОГА не публикувай твърдения за конкретни физически лица, непроверими обвинения
  или политически внушения. Промптите на министрите съдържат тази забрана изрично.
- Сигнали от граждани не се публикуват сурови — само анонимизирана, агрегирана статистика.
- Scraping: уважавай robots.txt, rate limit ≥ 1.5s между заявки, коректен User-Agent
  с контакт. Само публично достъпни данни.
- При съмнителен източник или данни — маркирай несигурността в отчета, не я скривай.

## Чл. 6 — Качество на кода

- Python 3.12, uv, ruff, mypy (strict за `core/`), pytest. Тестове за всеки договор
  и за опашката. e2e тестът ползва `tests/fake_brain.py`, никога реален LLM.
- Разработка и тестове: винаги `--dry-run` / fake brain. Реалният `claude -p` се вика
  само от насрочените сесии на кабинета, за да се пази Pro лимитът.
- Код и README — английски. Отчети, промпти на министрите, сайт — български.

## Чл. 7 — Работен ритъм

- `aigov ingest` тече на час (systemd timer) — безплатно, чист Python.
- Сесия на кабинета: 2x дневно, обработва всички pending задачи в ЕДИН пакетен run.
- След сесия: `core/publish` → `site/build.sh` → deploy. Провалени задачи отиват
  във `failed/` с причина и се докладват в лога, не блокират останалите.
